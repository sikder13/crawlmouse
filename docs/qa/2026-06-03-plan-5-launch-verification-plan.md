## Governing principles (the safety + objectivity contract)

1. **Unit / deterministic tests are the primary coverage of all *logic*.** They are re-runnable, prove exact values, and touch nothing in prod. Almost every Phase 0→6 invariant already has a pure helper with a green vitest suite (cited per-TC); the deterministic layer **runs + asserts those suites by exact count**, and adds the few cases that are missing. The live layer verifies ONLY what a unit cannot: the real SSE wire, the real browser render, the real PostHog/Sentry sink, the real RLS grant, and live config.
2. **Full-table crons are NEVER invoked live/unscoped against prod.** The scheduled reconcile cron iterates EVERY customer and writes `pro_until`; the TTL-cleanup cron deletes over the whole `audits` table. These are proven by their (shared) pure helpers `runReconcile` / `reconcileCustomerChunk` / `deleteExpiredAudits` (`inngest/billing-helpers.ts`) under a **mocked DB**, plus read-only blast-radius counts on prod. The ONLY thing that may touch prod for billing is the **DRY-RUN** reconcile (`runReconcile(..., {mode:'dry-run'})`) which by construction writes nothing — proven by a read-only before/after `pro_until` snapshot showing **0 changed rows**. Any genuinely-live *seeded* full-run (drift-repair, batched delete) is exercised ONLY on an **isolated, disposable** Supabase dev branch (`mcp__claude_ai_Supabase__create_branch` → seed → exercise → `delete_branch`) or a local supabase stack — **never** project `ezspnfeyzwsisymytssm`.
3. **No blocked DML on prod.** The Supabase MCP auto-classifier blocks unscoped INSERT/UPDATE/DELETE. Every precondition on prod is established through the **real app path** (`POST /api/audits/start`, a magic-link sign-in, the admin takedown route) — never by direct seeding. Every prod DB *assertion* is read-only `execute_sql` (SELECT). Every destructive cleanup is an **equality/key-scoped** DELETE — `id`-scoped where an `id` column exists (the classifier permits `id`-scoped DELETE), else an exact equality predicate on the row's unique key (`bucket_key` for `rate_limits`, which has **no `id`**; `lower(email)+source` for `waitlist`, whose unique index is the functional `lower(email),source` expression). Because the MCP auto-classifier's acceptance of an equality predicate on a NON-`id` column is unverified, every such non-`id` DELETE has an explicit fallback: **if the classifier rejects it as "unscoped", the user runs the identical equality-scoped statement in the Supabase SQL editor** (never widened to a wildcard). So cleanup re-runnability never silently depends on an unverified classifier behavior. **The one narrow, documented exception** to "no prod write": IF a same-day re-run forces a rate-limit-bucket reset, it is preferably done on the disposable DEVBRANCH; if done on prod at all it is limited to an **equality-scoped** DELETE of the per-run TEST-NET-3 `ip:203.0.113.<n>` / `waitlist:ip:203.0.113.<n>` keys ONLY (predicate column `bucket_key`, the `rate_limits` PK has no `id` — see Test identities). The shared singleton `global:audits:day` row is **NEVER** deleted on prod under any circumstance.
4. **Every pass criterion is a single computed value, HTTP status, or count** — derived from the code's own constant/formula (cited as `file:symbol`), never "looks right". Where a value comes from a constant, the TC cites the constant (`FREE_FINDING_LIMIT=5`, `GLOBAL_AUDITS_PER_DAY=5000`, `WAITLIST_PER_IP_PER_DAY=5`, `FREE_PAGE_CAP=500`/`PRO_PAGE_CAP=2000`, `FREE_CONCURRENCY=1`/`PRO_CONCURRENCY=8`, `AUTOCAPTURE_SAMPLE_RATE=0.1`).
5. **No silent skips.** Any TC that can't run in this environment (k6 absent; a full-table cron; a headless-unseedable Pro state) is explicitly mapped to its deterministic unit coverage and/or an isolated-scope substitute — recorded as "covered by X", never dropped.
6. **Explicit evidence + cleanup.** Each TC writes under `evidence/plan-5/`. Durable ids (audit ids, the dev-branch id, any auth uid) go to `evidence/plan-5/manifest.env`; §C cleanup `source`s it so completeness is mechanical, not eyeballed.
