## FINDING (live-only; unit tests can't see it) — spurious wouldRepair from a pro_until STRING-format mismatch  [LOW/MINOR]
The dry-run logged `would set 67977a90… pro_until=2026-07-06T04:21:20.000Z` and counted wouldRepair:1 even though
AU1's pro_until is ALREADY correct. Cause: runReconcile compares `(row.pro_until) !== derived.proUntil` as STRINGS
(billing-helpers.ts:170). The DB returns timestamptz as `2026-07-06T04:21:20+00:00`, while derived.proUntil is
`new Date(periodEnd*1000).toISOString()` = `2026-07-06T04:21:20.000Z` — the SAME INSTANT, different string format
(`+00:00`/no-ms vs `.000Z`). So the comparison is always true for active subscribers.
  IMPACT: (a) the dry-run/ops `wouldRepair` metric is inflated — every active sub looks "drifted", making the metric
  useless for "how many actually drifted?"; (b) a FULL reconcile would do a harmless NO-OP UPDATE (rewrites the same
  instant) for every active sub each run. NO entitlement impact, NO data corruption (the instant is identical).
  WHY UNIT TESTS MISS IT: TC-A1/A2/A3 compare ISO-string to ISO-string (iso(5000) on both sides) so the formats match;
  only a real timestamptz round-trip exposes the mismatch.
  DEPLOY-GATE / fix (§D): compare pro_until by INSTANT (e.g. `new Date(a).getTime() === new Date(b).getTime()`, or
  canonicalize both sides) instead of raw string equality, at billing-helpers.ts:170 (and the dry-run log at :172).
  Recommend a follow-up fix-PR before relying on the reconcile drift metric in ops.
